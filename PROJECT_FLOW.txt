PROJECT FLOW - NextHire AI

1. PROJECT OVERVIEW

NextHire AI is a combined web application for resume building, ATS (Applicant Tracking System) scoring, and AI-driven interview practice. It provides:
- A Resume Builder with multiple HTML/CSS templates and PDF output
- An ATS scoring and recommendation engine that compares resumes to job descriptions
- An AI Interview Simulator that generates adaptive interview questions and evaluates answers
- Voice-enabled interview mode (speech-to-text + text-to-speech)
- Integration with the Gemini API for question generation and evaluation

Primary technologies used (as observed in the codebase):
- Frontend: React (Vite) — pages include `ResumeBuilder.jsx`, `InterviewPrep.jsx` and components like `LivePreview.jsx` and `TemplateSelector.jsx`.
- Backend: Flask (Python) — `backend/app.py` exposes API endpoints for resume handling, ATS scoring, interview generation/evaluation, and PDF generation.
- Gemini API: used as the LLM engine via an internal wrapper `_gemini_generate_text()` (backend) to generate questions and evaluate answers.
- PDF generation: The backend converts rendered HTML (Jinja templates in `backend/templates/*`) into PDF. Current implementation uses WeasyPrint in `_html_to_pdf_bytes`, but the architecture references common tooling such as `pdfkit`/`wkhtmltopdf` as alternative approaches.
- Web Speech APIs: `SpeechRecognition` (browser-side, often `webkitSpeechRecognition`) for capture/transcription and `SpeechSynthesis` for AI voice playback.


2. ATS SCORE SYSTEM FLOW

Overview
The ATS Score System measures how well a candidate's resume aligns with a provided job description (JD). It extracts keywords/skills/technologies from both resume and JD, computes semantic similarity, and synthesizes a final ATS score along with recommendations.

High-level ATS API flow
- Frontend: user uploads or pastes a resume and enters/pastes a job description.
- Frontend → Backend: POST request with `resume_text` and `job_description` (or resume upload endpoint that parses PDF/DOCX into text).
- Backend: normalize and tokenize both texts, extract keywords and skill tokens, compute match metrics and semantic similarity.
- Backend → Frontend: return structured ATS result including `ats_score`, breakdown (keyword_score, semantic_score, skill_score), matched/missing keywords, and suggestions.

Key backend components and logic
- Text extraction: If the user uploads a PDF/DOCX, server-side parsers extract the resume text (endpoints: `/api/upload-resume` and other resume endpoints). Extractors are implemented in helper functions (PDF/DOCX parsing utilities). For pasted text, the request carries the raw `resume_text`.

- Tokenization and keyword extraction:
  - The backend tokenizes input, removes stopwords, and extracts candidate keywords and phrases (helper functions: tokenizers and `_extract_keywords()`-style utilities).
  - Keywords and tokens from both JD and resume are represented as sets for matching.

- Scoring components (as implemented):
  - `keyword_score`: proportion of JD keywords that appear in the resume keywords (expressed as percentage).
  - `semantic_score`: semantic similarity between resume and JD, computed using an embeddings model when available (`_semantic_similarity_score()` uses model.encode and cosine similarity). If embeddings are unavailable, a fallback cosine similarity on engineered features is used.
  - `skill_score`: overlap of curated skill tokens (common skills or phrase skills) between resume and JD. If JD has identified target skills, skill_score measures coverage of those by the resume.

- Final ATS aggregation:
  - The final `ats_score` is a weighted aggregate, typically: 0.4 * keyword_score + 0.4 * semantic_score + 0.2 * skill_score. (This weight breakdown is used in the code.)
  - Scores are clamped to sensible ranges (e.g., 0-100) and rounded.

- Missing skills detection & recommendations:
  - Matched keywords are recorded. Missing keywords (JD keywords not present in resume) are listed.
  - Suggestions are heuristics-driven (e.g., if resume token length is short relative to JD, suggest improving experience section; if semantic_score low, suggest improving project descriptions).
  - A de-duplication pass removes duplicate suggestions.

- Confidence and explanation:
  - Confidence can be interpreted from semantic_score and keyword coverage; backend provides `semantic_score`, `keyword_score`, and `skill_score` so the frontend can show confidence cues.

Example API endpoints and flows
- Resume upload: `POST /api/upload-resume` → parses and returns `parsed_text`.
- ATS calculation / resume analysis flows: (internal endpoints used by frontend) send `resume_text`, `job_description`, and optional metadata. The backend returns JSON with `ats_score`, `matched_keywords`, `missing_keywords`, `suggestions` and other telemetry.

Notes on Gemini involvement
- The ATS primarily uses deterministic NLP (tokenization, keywords, embeddings). Gemini may be used in some flows (e.g., generating tailored resume suggestions or improving summaries) via endpoints like `/api/generate-summary` or `/api/improve-text`.


3. AI INTERVIEW SYSTEM FLOW

Overview
The AI Interview System runs an adaptive interview: it generates a single follow-up question at a time, evaluates the candidate's answer, records the turn (question + answer + feedback), and uses the accumulated history to steer the next question. This is implemented across frontend interview UI (`InterviewPrep.jsx`) and backend generation/evaluation endpoints (`/api/generate-question`, `/api/evaluate-answer`, `/api/interview-summary`).

3.1 Interview Setup (Frontend)
- Users choose:
  - Job role (preset or custom)
  - Experience level (Beginner / Intermediate / Advanced)
  - Interview type (Technical / HR / Behavioral / Mixed)
  - Number of questions for the session (e.g., 5/10/15)
  - Mode: TEXT or VOICE
- Initial state: `turns` (local list of question/answer/feedback), `questionHistory`, and `coveredConcepts` keep track of Q&A and derived concepts.

3.2 Question Generation (Backend via Gemini)
Flow
- Every request for a new question is: Frontend POST to `/api/generate-question` with a payload containing the session settings and the interview history. Fields include: `role`, `experience_level`, `interview_type`, `question_count`, `question_number`, `interview_history` (structured sequence of prior turns), and `previous_questions` (array of questions asked so far). We also send `covered_concepts` (extracted on the frontend) for additional anti-repeat guidance.

Backend logic (question generation)
- The server normalizes and builds a `history_prompt` summarizing the last N turns (e.g., last 8).
- The backend composes a detailed Gemini prompt that includes the role, experience level, interview type, question number, adaptive guidance (derived from prior scores), parsed resume context (if available), the full prior Q&A, a clear "PREVIOUS QUESTIONS ASKED" list and a "CONCEPTS ALREADY COVERED" list, and an explicit anti-repeat rule set:
  - NEVER repeat the same question or paraphrase it
  - NEVER ask about a concept already covered unless doing a deep follow-up
  - Prefer new angles, new concepts or deeper follow-ups based on performance trend
- The model is instructed to return strictly valid JSON containing keys: `question`, `difficulty`, `focus_area`, `follow_up_style`.

Anti-repeat mechanisms
- Prompt-level: the Gemini prompt explicitly lists previous questions and concepts and instructs the model to avoid repeats.
- Backend-level: after Gemini responds, the backend extracts JSON and applies additional duplicate detection heuristics:
  - Normalize question text (strip punctuation, lower-case, condense whitespace)
  - Compare normalized signatures to previously asked questions
  - For near-duplicates, use fuzzy string similarity (difflib.SequenceMatcher) with a high threshold (e.g., >= 0.88) to detect paraphrases and reject repeated outputs
- If Gemini fails or returns a duplicate, backend falls back to `_fallback_question()` which picks a context-aware question from a trend-based pool and avoids previously used signatures if possible.

Adaptive question sequencing and trend detection
- The backend calculates a `trend` ("easier", "balanced", "harder") by summarizing scores from the normalized interview history (`_history_summary()`):
  - If average score < 4 -> easier
  - If average score 4-7 -> balanced
  - If average score >= 7 -> harder
- This trend is passed into the prompt to bias Gemini's follow-up style and content.
- The backend also creates a short `guidance` string describing the candidate's recent strengths/weaknesses to guide the model.

3.3 Interview Evaluation (Backend via Gemini)
Flow
- After the user answers a question (or the voice transcript is captured), the frontend POSTs to `/api/evaluate-answer` with: `role`, `experience_level`, `interview_type`, `current_question`, `user_answer`, and `interview_history`.

Evaluation prompt and expected output
- The backend composes a long evaluation prompt instructing Gemini to act as a rigorous hiring manager and produce strictly structured JSON with keys like `score` (1-10), `strengths`, `weaknesses`, `suggestions`, `ideal_answer`, `communication_rating`, `technical_rating`, `confidence_rating`, `performance_level`, `follow_up_direction`, and `short_feedback`.
- The prompt includes the role, experience level, question number, the candidate's answer, resume context (if provided), interview history, and previously covered concepts.

Feedback normalization and fallback
- The backend attempts to parse the LLM JSON response; if parsing fails or Gemini returns incomplete output, the server falls back to a local heuristic evaluator (`_fallback_evaluation()`), which computes a base score from answer length and presence of structured elements (metrics, examples, STAR structure), and produces example strengths/weaknesses and suggestions.
- The normalized feedback is attached to the `turn` saved by the frontend and used to compute `summaryData` (aggregate scores, strengths, areas needing improvement).

3.4 Frontend turn flow
- `Start Interview` → calls `/api/generate-question` to retrieve first question.
- User answers → frontend POSTs to `/api/evaluate-answer` and receives feedback.
- Frontend appends the turn (question, answer, feedback) to local `turns`, extracts concepts from the turn to update `coveredConcepts`, and stores question text into `questionHistory`.
- `Next Question` triggers `/api/generate-question` again with the updated `interview_history`, `previous_questions`, and `covered_concepts`.
- At session end or when question count is reached, frontend calls `/api/interview-summary` for a final summary.

Important guarantees implemented
- NEVER repeat any question or concept: enforced both in prompt and with backend duplicate detection.
- All Gemini requests include prior Q&A history and covered concepts to inform generation.
- Interview questions are single, focused, JSON-only outputs from the LLM to simplify parsing and robust behavior.


4. VOICE INTERVIEW FLOW

Overview
Voice mode enables spoken answers and AI voice playback. The frontend uses the browser SpeechRecognition API to transcribe microphone input and SpeechSynthesis API to read AI questions aloud.

Detailed flow
- On receiving a question (either initial or subsequent), the frontend sets `currentQuestion` and, if voice is enabled, calls SpeechSynthesis (`SpeechSynthesisUtterance`) to speak the question aloud via the selected browser voice.
- The user toggles recording (or in a voice-only mode, recording auto-starts). The app creates a `SpeechRecognition` instance (window.SpeechRecognition or webkit variant) and listens for speech.
- Recognition behavior:
  - `interimResults` = true and `continuous` = true ensure partial transcripts appear while the user speaks.
  - The app maintains a silence timer: when no new interim/final result has been captured for ~2 seconds, it auto-stops recognition and finalizes the `liveTranscript`.
  - The final transcript is stored as the user's answer (`answer`) and submitted to `/api/evaluate-answer` just like typed text.

Why audio → text before Gemini
- Gemini (LLM) operates on text. Converting audio to text allows the same evaluation/generation logic to be reused for both text and voice modes.
- Transcription ensures deterministic and parseable input for scoring and avoids adding an extra layer of audio-model complexity.

Edge cases and UX details
- The UI disables text inputs in voice mode to avoid conflicts.
- The system handles recognition errors gracefully and provides user-facing messages.



7. GEMINI API INTEGRATION

High-level integration
- The backend exposes an internal helper `_gemini_generate_text(prompt: str) -> str` which calls out to the configured LLM (Gemini) and returns the raw text output.
- For structured tasks, the backend sends detailed instructions and demands strictly JSON-only outputs. Post-response, the code uses `_extract_json_object()` to strip code fences and extract an inline JSON object from the model output.

Request structure and prompt patterns
- Question generation prompts include:
  - Session settings (role, experience, type, question number/total)
  - Adaptive guidance and trend (derived from history)
  - Resume context preview (if available) — but this project removes CV-derived prompts from interview generation when CV upload is disabled by policy; otherwise, the parsed resume preview can be included truncated to a safe size.
  - Interview history (formatted concise lines, most recent last).
  - An explicit list: "PREVIOUS QUESTIONS ASKED" and "CONCEPTS ALREADY COVERED".
  - A list of "CRITICAL ANTI-REPETITION RULES" instructing the model to never repeat or paraphrase earlier questions.
  - Output instructions: "Return ONLY JSON with keys: question, difficulty, focus_area, follow_up_style".

- Evaluation prompts include:
  - Role, experience, interview type, question number/total
  - The exact question and the candidate's answer
  - Resume context (if provided)
  - Interview history and previously covered concepts
  - A strict evaluation rubric describing desired JSON output keys (score 1-10, ratings 1-10, arrays for strengths/weaknesses/suggestions, and `short_feedback`), plus role-specific evaluation criteria.

Response parsing and error handling
- The backend extracts the JSON object using `_extract_json_object()` which strips fences and locates the JSON blob.
- If parsing fails, or if the model returns repeated or otherwise invalid content, backend falls back to deterministic fallback functions: `_fallback_question()` for question generation or `_fallback_evaluation()` for scoring.

Context and adaptive management
- The backend always sends prior Q&A (history) in Gemini requests — the request includes the normalized interview history summary and the last ~N turns for safety & brevity.
- The frontend also sends arrays like `previous_questions` and `covered_concepts` to make it easier for the backend prompt construction.

Example API flow for question generation
- Frontend POST to `/api/generate-question` with `{ role, experience_level, interview_type, question_count, question_number, interview_history, previous_questions, covered_concepts }`.
- Backend builds the prompt, calls `_gemini_generate_text(prompt)`, extracts JSON, performs duplicate checks, and returns `{ question, difficulty, focus_area, follow_up_style, trend, question_number, total_questions }`.


8. AI LOGIC EXPLANATION

Adaptive behavior and recruiter simulation
- The system simulates a recruiter by:
  - Using role and interview type as the framing context
  - Summarizing prior turns to calculate candidate trend (easier/maintain/harder)
  - Including recent weaknesses/strengths in the prompt so the model can focus follow-ups on gaps or build on strengths
  - Enforcing anti-repetition: both in-prompt rules and backend duplicate detection ensure questions are new and adapted

Follow-up question strategies
- If the candidate scores high: requests for deeper reasoning, trade-offs, architecture, and edge-case analysis are preferred.
- If the candidate scores mid-range: ask related but different concepts to probe breadth.
- If the candidate scores low: ask simpler, fundamentals-focused questions to build confidence and surface gaps.

Non-repetition guarantees
- Prompt-level explicit lists and rules ("NEVER repeat or paraphrase")
- Backend-level normalization & fuzzy matching to catch paraphrases (signature normalization + similarity ratio), rejecting near-duplicates and falling back to varied alternatives.

How the recruiter persona is implemented
- The Gemini prompts are authored to adopt a professional, realistic interviewer tone and return a single follow-up question. The evaluation prompt asks the model to behave like a senior hiring manager and produce actionable feedback, not generic phrases.

11. CONCLUSION

This document describes the internal flows and integration points for NextHire AI. The system is designed to keep the user experience consistent and deterministic by using server-side template rendering for the resume builder preview and PDF generation, and by enforcing robust context passing and anti-repetition strategies for the AI interview flows. The Gemini API is used as the primary LLM engine for question generation and evaluation; fallback heuristics ensure resilience if the LLM output fails validation.

For evaluation or presentation, reviewers can exercise the main flows:
- Build a resume, preview it, and generate a PDF to verify server-rendered fidelity.
- Run an interview session in text and voice mode to observe adaptive questioning and non-repetition.
- Provide a job description and sample resume text to exercise the ATS scoring and recommendations.
